# 3. Credibility corrections and public-claim safety

## 3.1 H3+, Q20+ and internal-tier language

Use **H3+** and **Q20+** as internal build labels inside project documentation, not public claims. There is public documentation about xAI's Colossus cluster and public documentation about Google Cloud resource quotas and MLOps, but there is no reliable public basis to claim personal access to xAI Horizon infrastructure, Google internal Q20+ compute allocations, or internal Google/xAI engineering review processes.

Correct public wording:

> Built with scale-ready engineering patterns inspired by frontier AI infrastructure: sharded training skeletons, columnar data pipelines, CI-gated evaluation, drift checks, observability, privacy controls, and reproducible deployment artifacts.

Incorrect public wording:

> Built to xAI Horizon 3+ and Google Q20+ standards.

The second sentence sounds like a false credential. Do not use it publicly.

## 3.2 Level 3 MLOps correction

Public Google Cloud MLOps material commonly describes maturity levels such as manual process, pipeline automation, and CI/CD pipeline automation. If this dossier uses **Level 3+ MLOps**, it means a **portfolio-defined extension** beyond the public maturity model:

```text
MLOps-L0: Manual notebook / ad hoc work
MLOps-L1: Repeatable scripts + tests
MLOps-L2: CI/CD + automated evaluation
MLOps-L3+: Continuous data validation + drift checks + evaluation gates + rollback policy + reproducible evidence
```

Do not imply this is an official Google "Level 3" category unless a public source is cited and wording is exact.

## 3.3 SRE correction

Do not claim 99.99% availability for free-tier demos. Free-tier platforms do not give you enterprise SLA guarantees. Instead, demonstrate SRE thinking:

- Define SLIs.
- Define SLO targets.
- Add synthetic checks.
- Add latency tests.
- Add error-budget-style launch gates.
- Add fallback states when demos sleep or rate-limit.
- Publish a launch report with limitations.

Correct wording:

> SRE-style operating discipline with documented SLIs, SLO targets, fallback states, and synthetic checks.

Incorrect wording:

> This free-tier portfolio has Tier 1 Google SRE availability.

# 4. Enterprise levels translated into zero-cost proof

## 4.1 Compute & Training Scale: H3+/Q20+ style proof

### Enterprise meaning

Large training systems use distributed compute, memory sharding, data-parallelism, pipeline-parallelism, high-throughput data loaders, observability, and reproducible experiments.

### Zero-cost portfolio translation

You do not need frontier compute. You need code that proves you understand scale constraints.

Required proof artifacts:

| Proof artifact | Repository location | Why it matters |
|---|---|---|
| FSDP or DeepSpeed skeleton | `training/fsdp_train.py` or `training/deepspeed_train.py` | Shows distributed-training literacy |
| Small local smoke test | `tests/test_training_smoke.py` | Avoids dead enterprise theatre |
| Config-driven training | `configs/train.yaml` | Shows reproducibility |
| Model card | `MODEL_CARD.md` | Shows responsible ML practice |
| Resource profile | `reports/resource_profile.md` | Shows memory/latency awareness |
| Quantization note | `reports/quantization_report.md` | Shows zero-cost serving optimization |

### Implementation principle

Do not force distributed training into projects that do not need it. Use it where it makes sense:

- VCCF/RAG: optional embedding or reranker fine-tuning skeleton.
- AI demo: small transformer classifier fine-tune skeleton.
- Dissertation Random Forest: do not pretend FSDP applies. Use Ray/Arrow/DuckDB/Parquet scale-readiness instead.

## 4.2 Architecture Maturity: MLOps-L3+ proof

Required proof artifacts:

| Proof artifact | Repository location | Tool |
|---|---|---|
| Automated tests | `tests/` | pytest |
| Data validation | `quality/expectations/` | Great Expectations or custom checks |
| Drift checks | `monitoring/drift_report.py` | Evidently AI |
| CI workflow | `.github/workflows/qa.yml` | GitHub Actions |
| Model evaluation | `eval/evaluate.py` | sklearn / custom |
| Model registry metadata | `models/model_registry.json` | local JSON |
| Rollback policy | `docs/ROLLBACK_POLICY.md` | documentation |
| Launch report | `docs/LAUNCH_REPORT.md` | documentation |

## 4.3 Serving Infrastructure: Tier-1-style SRE proof

Required proof artifacts:

| Proof artifact | Repository location | Tool |
|---|---|---|
| Dockerfile | `Dockerfile` | Docker |
| Health endpoint | `/health` | FastAPI |

## 4.4 Observability: OpenTelemetry + three pillars

# 10. H3+ compute and scale-readiness implementation

## 10.1 What to prove

The goal is not to train a frontier model. The goal is to show that code can scale beyond a notebook.

Required patterns:

- Config-driven execution.
- Deterministic seeds.
- Dataset streaming or chunking.
- Columnar storage with Parquet / Arrow.
- Memory profiling.
- Optional Ray parallelism.
- Optional distributed training skeleton for deep-learning demos.
- Quantization report if hosting a model.
- Small smoke tests that run locally.

## 10.2 Data pipeline scale pattern

Use this pattern for dissertation and data-quality projects:

```text
CSV / OSM / boundary files
  -> schema validation
  -> PII scan if applicable
  -> Arrow/Parquet intermediate
  -> chunked transforms
  -> quality checks
  -> feature table
  -> model / dashboard output
  -> QA report
```

## 10.3 Recommended local tools

| Need | Tool | Why |
|---|---|---|
| Columnar data | Apache Arrow / Parquet | Analytics-friendly, zero-copy concepts, efficient scans |
| Local SQL analytics | DuckDB | Fast local analytical database |
| Larger-than-RAM processing | Ray Core or Dask | Shows scale-readiness |
| Geospatial pipeline | GeoPandas + pyogrio + Parquet | Fits dissertation project |
| ML evaluation | scikit-learn | Fits UK data analyst/junior DS roles |

def run_drift_check(reference_path: str, current_path: str, report_path: str) -> None:

    # Replace with Evidently report generation in implementation.

import torch
import torch.distributed as dist
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP


def setup_distributed():
    if not dist.is_initialized():
        dist.init_process_group("nccl")
    torch.cuda.set_device(dist.get_rank() % torch.cuda.device_count())


def train(model, dataloader, optimizer, device):
    model.train()
    for batch in dataloader:
        optimizer.zero_grad(set_to_none=True)
        inputs, labels = batch
        inputs = inputs.to(device)
        labels = labels.to(device)
        loss = model(inputs, labels=labels).loss
        loss.backward()
        optimizer.step()


def main():
